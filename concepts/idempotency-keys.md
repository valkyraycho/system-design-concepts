# Idempotency keys & exactly-once effect

## The mechanism

Client attaches a **unique key per logical intent** (UUID); server remembers processed keys and **replays the stored result** on a repeat instead of re-executing.

- Key generated by the **client, once per intent, reused across retries.** New key on retry = back to double-charging. The key = "intent to do this one thing," NOT "this network attempt."

## Must be atomic — TOCTOU hides inside the mechanism

Naive "GET key → if absent, process → SET key" double-charges on concurrent retries (both GET miss before either SETs). **Storage per-command atomicity (Redis single-threaded) does NOT cover your check-then-act gap.**

Fix: **atomic check-and-set** — Redis `SET key val NX`, or SQL unique-constraint insert that rejects the duplicate. Pattern:
1. Atomically insert key as `in_progress` (2nd concurrent attempt fails the unique insert)
2. Winner does work → update to `completed` + store result
3. Repeat sees key: `completed` → return stored result; `in_progress` → "still processing, retry later"

**Idempotency is built ON TOP OF atomicity** (the uniqueness constraint makes the "seen it?" check safe). They cooperate: atomicity = primitive; idempotency = pattern using it + stored results.

> Same TOCTOU bug, three costumes now: seat oversell, idempotency-key race, here. "Storage is atomic so I'm safe" is false unless the *entire* check-and-act is ONE atomic op (SET NX / INCR / Lua / txn).

## Design questions (Stripe case study)

- **Key TTL** — forever = unbounded growth; too short = delayed retry double-charges. Stripe ~24h. (Same window-tuning shape as token lifetime, backup retention.)
- **Same key, different body** → reject (key promises identical request). Stripe fingerprints the request.
- **Stuck `in_progress`** (crash mid-op) → make the underlying op itself safe (e.g. the bank charge uses its *own* idempotency key).

## Dual-write problem & exactly-once EFFECT

You can't atomically do a DB change AND an external action (charge in DB, email at SendGrid — no shared transaction). Any crash in the gap → inconsistency. **No perfect fix.**

**Key reframe:** you can't get exactly-once *delivery*; you get exactly-once **effect** = at-least-once delivery + idempotent dedupe. "Exactly once" is a property of the *outcome*, manufactured from unreliable at-least-once parts.

**Transactional outbox** (Ray reinvented this — "a column indicating email sent"):
1. In the **same transaction** as the charge, insert an `outbox` row ("send receipt for order 123") → charge + intent-to-email commit atomically (closes the dual-write gap)
2. Separate worker reads outbox, sends, marks `sent`
3. Crash after send before mark → re-send (at-least-once) + dedupe key at provider → user sees one

## Match rigor to stakes (senior judgment, Ray's instinct)

Full outbox+dedupe for the **charge** (money → exactly-once effect required). For a **receipt email**, at-least-once with rare duplicates is usually fine (annoying, not a financial incident). Don't build exactly-once infra for a notification nobody reconciles.
